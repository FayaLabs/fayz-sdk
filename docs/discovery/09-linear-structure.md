# 09 — Proposed Linear Structure

This is the proposed tracking structure before mutating Linear.

## Parent epic

**Title:** Epic: Fayz SDK inside Fayz — manifest Panel, tenant SaaS surfaces, plugins, and agent-ready project scaffolds

**Project:** `fayz.ai`

**Cycle:** 21 (`2026-06-15` → `2026-06-22`)

**Labels:** `epic`, plus owner labels as appropriate.

**Description summary:**

Create the first integrated Fayz SDK foundation inside Fayz: tenant-specific manifest rendering in the editor Panel, provider abstraction for Supabase/Fayz API/custom providers, generated-project SDK scaffolding, agent instructions, retro-compat import of SDK apps, and validation through Beauty SaaS demo usability.

**Route recalculation:** this epic is now proof-first. Public npm surface is intentionally limited to `@fayz-ai/sdk`. App runtime, UI, SaaS, shop, storefront, portal, courses, and plugin packages are internal/private until Beauty plus a second vertical prove stable seams. Linear children should be written as user stories for real app outcomes, not as package-publication work.

**Historical origin:** FAY-924.

## Child issue 1 — Manifest/provider contract

**Title:** Lock AppManifest + provider contract for tenant-specific Fayz Panel rendering

**Outcome:** One manifest shape and provider abstraction supports Supabase default, Fayz API provider, and future custom providers.

**Dependencies:** none.

**Acceptance criteria:**

- `BackendRef`/provider strategy supports Fayz API without hard-coding Supabase.
- App/Surface manifest model is documented as the canonical Panel rendering contract.
- Validation catches missing/invalid surfaces/pages/plugins/provider refs.
- Beauty SaaS has a first extracted/representative manifest fixture.

## Child issue 2 — Fayz DB/API manifest resolver for Panel

**Title:** Store and resolve per-project/per-tenant Panel manifest in Fayz API

**Outcome:** Fayz editor can ask API for the manifest for a project/customer/tenant.

**Dependencies:** issue 1.

**Acceptance criteria:**

- DB table/model name is locked and migrated.
- API endpoint returns manifest + invariant Panel features.
- Resolver handles project/tenant/customer identity explicitly.
- Test covers two tenants with different manifests.

## Child issue 3 — Render SDK surface in Fayz editor Panel

**Title:** Render Fayz SDK manifest surface inside editor Panel tab

**Outcome:** Panel tab renders pages/plugins from manifest JSON.

**Dependencies:** issues 1 and 2.

**Acceptance criteria:**

- Panel renders admin/surface from returned manifest.
- Cloud Features/invariant entries remain visible.
- Manifest-only edit changes visible Panel output.
- No optimistic/frontend-only fake state for backend-authored content.

## Child issue 4 — Generated project scaffold includes Fayz SDK

**Title:** Add Fayz SDK package/context to new Fayz project scaffolds

**Outcome:** New projects generated by Fayz include SDK dependencies, manifest file, and agent-readable usage docs.

**Dependencies:** issue 1.

**Acceptance criteria:**

- Project package generation includes SDK dependency/workspace shape.
- Template includes `app.manifest.json` or equivalent starter manifest.
- Template includes agent guide for SDK/plugin/module usage.
- Build of generated project still passes.

## Child issue 5 — Agent guide for SDK modules/plugins/migrations

**Title:** Teach coding agents how to use Fayz SDK modules, plugins, and migrations

**Outcome:** Claude/Fayz agents can safely add a module, page, provider, and migration.

**Dependencies:** issues 1 and 4.

**Acceptance criteria:**

- Agent guide explains core settings, installing plugin/module, adding custom code via registry, and running migrations.
- Example: add ecommerce/shop module and page.
- Guide includes non-goals and safety rules.

## Child issue 6 — Beauty SaaS retro-compat/import proof

**Title:** Import/migrate Beauty SaaS as Fayz project and prove manifest-driven changes

**Outcome:** Beauty SaaS is usable as the main proof vertical.

**Dependencies:** issues 1–4.

**Acceptance criteria:**

- Beauty SaaS represented/imported as a Fayz project.
- Agenda opens and can create booking.
- Manifest-only change modifies UI/Panel output.
- Feature gaps from Beautyplace are listed and prioritized.

## Child issue 7 — Tannat/store + shop module database proof

**Title:** Migrate Tannat Store and resolve shop module data model

**Outcome:** Ecommerce/storefront path proves SDK is not salon-only.

**Dependencies:** issue 1; preferably after Beauty proof.

**Acceptance criteria:**

- Tannat Store represented/imported as Fayz project.
- Shop module DB model is documented/normalized.
- Manifest/plugin structure supports store pages and products.

## Child issue 8 — Calendar and ecommerce module references

**Dependencies:** issue 1.

**Acceptance criteria:**

## Child issue 9 — Foundation hardening/design-system lock

**Title:** Lock design-system variation strategy before SDK standardization

**Outcome:** SDK package is safe to lock into all repos without brittle visual forks.

**Dependencies:** can run in parallel after issue 1.

**Acceptance criteria:**

- Theme/token variation strategy documented.
- Existing app variations mapped to supported levels.
- Build/typecheck/test gates pass across SDK package.

## Child issue 10 — E2E validation and demo script

**Title:** Validate Fayz SDK integrated demo across Fayz, generated project, and Beauty SaaS

**Outcome:** We can show customer-demo-grade proof.

**Dependencies:** issues 2–6.

**Acceptance criteria:**

- Fayz editor Panel renders SDK surface.
- Generated project includes SDK and builds.
