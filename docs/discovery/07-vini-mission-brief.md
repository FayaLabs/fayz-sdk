## Initial interpretation

This is not just “add SDK package.” It is a platform integration program with five coupled tracks:

1. **Manifest/tenant rendering** — Fayz editor Panel renders app/admin surfaces from per-tenant manifests.
2. **Provider abstraction** — SDK data access must support Supabase by default and Fayz API/custom providers as first-class adapters.
3. **Project scaffold integration** — every new Fayz-generated project includes SDK structure, package dependency, docs/context for coding agents, and migration path.
4. **Retro-compat + examples** — existing fayz-app projects become test fixtures and migration targets, not side demos.
5. **Plugin ecosystem foundations** — ecommerce/calendar are not random modules; they validate external-reference-inspired plugins and real business usefulness.

## Immediate architectural risk

Trying to implement all parts at once will create exactly the wrong foundation: half-runtime, half-plugin-marketplace, half-migration-system, and unstable app imports.

The safer order is:

1. lock the manifest and provider contracts;
2. make Panel render one real tenant manifest;
3. make project scaffold include the SDK contract;
4. import/convert one existing product (`beauty-saas`) end-to-end;
5. only then expand to Tannat/shop/calendar modules.

## Non-negotiables extracted

- Rendering must be tenant-specific and manifest-backed.
- Manifest must be stored/referenced in Fayz DB.
- Fayz SDK cannot assume Supabase only.
- Fayz-generated projects must include SDK by default.
- Agents must have explicit docs/context for using SDK and plugins.
- Existing SDK-consuming projects must be retro-compatible or migrated now.
- Beauty SaaS customer-demo usability is the minimum proof.
- Linear must track this as an epic/cycle-level program.

## Proposed KPI / validation signal

KPI: **SDK adoption proof** — number of real Fayz/Fayz-app projects that can render or be modified through SDK manifest/module definitions without bespoke code changes.

Minimum validation:

- `beauty-saas` imported/represented as a Fayz project;
- Panel renders its tenant-specific admin/pages from manifest;
- one booking can be created in agenda;
- at least one manifest-only change alters the rendered product;
- agent docs allow a coding agent to add/modify one plugin/module correctly.
